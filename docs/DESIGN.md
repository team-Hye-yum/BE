# DESIGN

백엔드 설계의 기준은 다음 사람이 안전하게 바꿀 수 있는 구조입니다.

## 설계 원칙

- 도메인 규칙은 흩어지지 않고 도메인 객체나 명확한 정책 컴포넌트에 위치합니다.
- Controller, Application/Service, Domain, Infra의 책임을 섞지 않습니다.
- 중복은 싫어하되 성급한 추상화는 더 경계합니다.
- 같은 정책이 세 번 이상 반복되고 변경 이유가 같을 때 공통화를 검토합니다.
- 공통화는 호출하는 쪽의 의도를 더 선명하게 만들어야 합니다.

## 좋은 설계 신호

- 유스케이스 흐름이 Service public 메서드에서 자연스럽게 읽힙니다.
- 실패 시 어떤 예외와 응답이 나가는지 추적할 수 있습니다.
- 테스트가 구현 세부가 아니라 행위를 검증합니다.
- 요구사항 변경 시 수정 위치가 예측 가능합니다.

## 나쁜 설계 신호

- Service 하나가 수천 줄로 커집니다.
- private 메서드가 계속 늘어나 절차형 스크립트가 됩니다.
- Entity가 getter/setter 묶음으로만 쓰입니다.
- `CommonUtils`가 도메인 의미를 흡수합니다.
- Controller가 Repository와 외부 API를 직접 호출합니다.
