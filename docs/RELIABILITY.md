# RELIABILITY

운영 안정성은 코드스타일의 일부입니다.

## 기본 기준

- 외부 API에는 timeout을 명시합니다.
- retry는 멱등성과 함께 설계합니다.
- DB 트랜잭션 안에서 외부 API 호출을 오래 붙잡지 않습니다.
- 이벤트 발행, 결제 승인, 메시지 전송은 실패 보상 전략을 검토합니다.
- N+1 가능성은 테스트나 SQL 로그로 확인합니다.
- health check와 metrics를 고려합니다.

## 장애 분석 가능성

- 요청 식별자, 사용자 ID, 도메인 ID를 로그에 남깁니다.
- 예외 로그는 스택트레이스를 보존합니다.
- 민감 정보는 로그와 오류 응답에서 제거합니다.
- 알 수 없는 예외는 내부 상세를 사용자 응답에 노출하지 않습니다.

## 검증 질문

- 장애가 나면 어떤 로그로 추적할 수 있는가?
- timeout, retry, circuit breaker가 필요한 외부 의존성이 있는가?
- 트랜잭션 범위가 장애 전파를 키우지 않는가?
- 테스트 환경이 운영 DB에 접속할 가능성은 없는가?

## DIVE 2026 데이터 신뢰성

- 전체 데이터는 샘플과 시트명, 컬럼명, 타입, 결측 구조가 다를 수 있습니다.
- 데이터 로딩은 실패 행 수와 실패 사유를 리포트해야 합니다.
- KODATA 조인 성공률을 매번 계산합니다.
- 2024년 지원은 후속연도 데이터가 없어 변화 확인이 불가할 수 있음을 정상 상태로 처리합니다.
- DB/서버 장애에 대비해 핵심 시연용 정적 JSON 폴백을 준비할 수 있게 설계합니다.
- 데이터 교체는 `docs/EXECUTION_RUNBOOK.md` 순서를 따르고, 결과는 `docs/DATA_QUALITY_REPORT.md` 형식으로 남깁니다.
