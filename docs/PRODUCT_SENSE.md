# PRODUCT_SENSE

제품 요구사항은 구현 지시가 아니라 수용 기준으로 바꿔 저장합니다.

## 요구사항 변환 원칙

- 사용자 피드백을 관찰 가능한 동작으로 바꿉니다.
- HTTP 상태, 오류 코드, 권한 조건, 데이터 노출 범위를 명시합니다.
- 실패 시 사용자에게 보일 메시지와 내부 로그에 남길 정보를 구분합니다.
- 목록/검색 요구사항은 성능과 조회 최적화 기준을 함께 둡니다.

## Codex에게 전달할 좋은 요청

- 어떤 사용자가 어떤 조건에서 무엇을 할 수 있어야 하는지 적습니다.
- 성공 응답과 실패 응답을 모두 적습니다.
- 검증해야 할 권한, 트랜잭션, 로그, 테스트 범위를 포함합니다.

## DIVE 2026 제품 판단 기준

- 담당자가 내일부터 실제 평가 업무에서 쓰고 싶은가?
- 평가 근거가 한 화면에서 더 빨리 이해되는가?
- 중복 지원과 유사 목적 반복 이력이 더 설명 가능해졌는가?
- 지원 이후 변화가 인과 주장 없이 정직하게 보이는가?
- 데이터 없음, KODATA 미매칭, 2024년 후속연도 부재가 버그처럼 보이지 않는가?
- 기업 대표가 화면을 보더라도 낙인처럼 느끼지 않는가?
