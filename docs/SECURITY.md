# SECURITY

보안은 Controller 내부 if문이 아니라 보안 설정, 서비스 경계, 테스트로 다룹니다.

## 기본 기준

- 인증과 인가를 구분합니다.
- URL 보안과 Method Security의 책임을 나눕니다.
- CSRF, CORS, 세션, 토큰 정책을 명시적으로 설정합니다.
- 비밀번호는 강한 해시 알고리즘으로 저장합니다.
- 권한 검사는 테스트합니다.
- 권한 문자열 비교가 여러 계층에 흩어지지 않게 합니다.

## 금지

- 비밀번호, 토큰, 주민번호, API key 등 민감 정보를 로그에 남기지 않습니다.
- 스택트레이스와 내부 상세를 API 응답에 포함하지 않습니다.
- 모든 예외를 HTTP 200으로 감싸지 않습니다.
- 인증 사용자 확인을 Controller마다 임의로 구현하지 않습니다.

## 리뷰 질문

- 이 API는 인증이 필요한가?
- 인증된 사용자가 해당 리소스에 접근할 권한이 있는가?
- 권한 실패는 `403`, 인증 실패는 `401`로 구분되는가?
- 민감 정보가 응답, 로그, 테스트 fixture에 남지 않는가?

## DIVE 2026 민감정보 기준

- KODATA 기업 신용, 재무, 고용 데이터는 해커톤 목적 한정 민감 정보입니다.
- 사업자등록번호, 법인등록번호, 대표자명, 기업명 원문은 비식별/가명 처리 대상입니다.
- `기업일련번호`는 결합키이지만 인증 수단이 아닙니다.
- 기업용 신청 전 점검 화면은 PoC이며 실제 본인 인증을 제공하지 않습니다.
- 식별번호만으로 조회되는 데모를 실제 접근 통제처럼 표현하지 않습니다.
- 원천 엑셀 행 전체를 API 응답이나 로그에 내보내지 않습니다.
